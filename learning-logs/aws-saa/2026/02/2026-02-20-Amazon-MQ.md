# 2026-02-20 — Amazon MQ (DevOps/SRE Lens)

## Core Mental Model

Amazon MQ = **Managed Message Broker for Legacy / Open Protocol Systems**

- SQS / SNS → Cloud-native messaging
- **Amazon MQ → Traditional message broker in AWS**

👉 Think:

- SQS = serverless queue  
- SNS = serverless pub/sub  
- **Amazon MQ = managed RabbitMQ / ActiveMQ**

---

# 1. Why Amazon MQ Exists

Problem:

Legacy systems use:

- AMQP  
- MQTT  
- STOMP  
- OpenWire  
- WebSocket (WSS)

These apps cannot easily migrate to:

- SQS
- SNS

Because:

- Protocol mismatch
- Application rewrite required

---

## Solution

Amazon MQ provides:

- Managed RabbitMQ
- Managed ActiveMQ

No code rewrite required.

---

# 2. Architecture Model

Applications → Amazon MQ Broker → Consumers

Broker supports:

- Queue model
- Topic model

In one service

---

# 3. Supported Protocols

Amazon MQ supports:

- AMQP
- MQTT
- STOMP
- OpenWire
- WebSocket (WSS)

### DevOps Insight

Amazon MQ fits:

- Hybrid architectures
- Lift-and-shift migrations
- Legacy enterprise apps

---

# 4. Scaling Model

Unlike:

- SQS
- SNS
- Kinesis

Amazon MQ:

- Runs on brokers (servers)
- Limited by instance size

### Tradeoff

| Service | Scaling |
|---------|--------|
| SQS | Infinite |
| SNS | Infinite |
| Kinesis | Very high |
| Amazon MQ | Broker-limited |

---

# 5. High Availability Architecture

Example

Region: us-east-1

Active:
- us-east-1a

Standby:
- us-east-1b

Storage:
- Amazon EFS

---

# 6. Failover Model

Active Broker → EFS  
Standby Broker → Same EFS

If Active fails:

- Standby takes over
- Clients reconnect
- Messages preserved

### SRE Insight

Amazon MQ uses:

- Shared storage
- Failover nodes
- Traditional HA architecture

---

# 7. Feature Comparison

| Feature | Amazon MQ | SQS | SNS |
|---------|-----------|-----|-----|
| Protocols | Open standards | AWS API | AWS API |
| Scaling | Limited | Infinite | Infinite |
| Queue | Yes | Yes | No |
| Pub/Sub | Yes | No | Yes |
| Serverless | No | Yes | Yes |
| Legacy migration | Yes | No | No |

---

# 8. When to Use Amazon MQ

Use Amazon MQ when:

- Migrating legacy systems
- Using AMQP / MQTT
- Need RabbitMQ features
- Enterprise messaging requirements

Avoid Amazon MQ when:

- Building new cloud-native apps
- Need infinite scaling
- Serverless preferred

---

# 9. Real-World Architecture

Legacy App → Amazon MQ →  
- Microservices  
- Worker nodes  
- Event processors  

---

# 10. DevOps/SRE Takeaways

Amazon MQ is:

- Managed broker
- Legacy-friendly
- HA-capable

But:

- Not infinitely scalable
- Requires capacity planning
- Traditional messaging model
