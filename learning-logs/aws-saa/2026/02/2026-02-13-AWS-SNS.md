# 2026-02-13 — Amazon SNS (DevOps/SRE Lens)

## Core Mental Model
SNS is not messaging.  
It is **event fan-out at scale**.

- One event → many consumers
- Push-based delivery
- Zero polling

👉 System shifts from:
**who should I call? → who is interested in this event?**

---

## 1. Architecture Primitive
**Topic → Publish → Fan-out → Subscribers**

- Topic = event hub
- Publisher = emits event
- Subscribers = receive independently

---

## 2. Why SNS Exists (Real Problem)
Without SNS:
- Producer must call every consumer
- Tight coupling
- Deployment friction
- Scaling complexity

With SNS:
- Producer sends **once**
- SNS distributes to **many**

👉 Insight:
SNS removes **N integrations → 1 integration**

---

## 3. Subscriber Model

### Direct Delivery
- Email / SMS / Mobile push
- HTTP/HTTPS endpoints

### AWS Integrations
- SQS → buffering layer
- Lambda → serverless processing
- Kinesis Firehose → analytics pipeline

👉 Common Pattern:
SNS → SQS → Workers  
(fan-out + buffering combined)

---

## 4. Delivery Model (Important)

- Push-based (no polling)
- Near real-time delivery
- At-least-once delivery

👉 SRE implication:
- consumers must handle duplicates
- failure handling is subscriber responsibility

---

## 5. Message Filtering (Underused Power)

- Subscribers define filter policies
- Receive only relevant events

### Example
- Topic = "orders"
- Subscriber A → only `status = shipped`
- Subscriber B → only `region = EU`

👉 Insight:
SNS can act as a **routing layer**, not just broadcast

---

## 6. Scale Characteristics

- 12M+ subscriptions per topic
- 100K topics per account

👉 Meaning:
SNS is designed for **mass fan-out**, not small systems

---

## 7. Event Sources (Native Producers)

- CloudWatch (alerts)
- Auto Scaling (lifecycle events)
- S3 (object events)
- RDS / DynamoDB / Lambda
- Budgets / CloudFormation

👉 Insight:
SNS is AWS’s **default event bus (lightweight)**

---

## 8. Security Model

- In-flight: HTTPS
- At-rest: KMS
- Access:
  - IAM (who can publish/subscribe)
  - Topic policies (cross-account, service integration)

---

## 9. Failure & Reliability Patterns

### Common Issues
- Subscriber endpoint down
- Message delivery failure
- Duplicate deliveries

### Mitigation
- SNS → SQS buffer pattern
- Retry policies
- Dead-letter queues (via SQS)

👉 Insight:
SNS alone is not durable buffering  
→ combine with SQS for resilience

---

## 10. SQS vs SNS (Clear Boundary)

| Service | Role |
|---|---|
| SQS | Buffer work (pull model) |
| SNS | Broadcast events (push model) |

👉 Combined Pattern:
- SNS = distribute
- SQS = absorb

---

## 11. Real Architecture Pattern

### Event-Driven System
1. Service emits event → SNS
2. SNS fans out:
   - SQS (worker processing)
   - Lambda (real-time logic)
   - HTTP (external systems)

👉 Result:
- loosely coupled system
- independent scaling
- failure isolation

---

## 12. Sharp Edges (Real Insights)

- No persistence guarantee like queues
- No ordering guarantee
- Subscriber failure handling is externalized
- Over-fanout can create hidden cost

---

## 13. One-Line Memory Anchor
> SNS = publish once, deliver everywhere.
