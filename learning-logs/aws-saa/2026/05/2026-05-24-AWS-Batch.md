# AWS Batch

---

# What is AWS Batch?

AWS Batch is a:

```text
Fully Managed Batch Processing Service
```

---

Purpose:

```text
Run Large Numbers of Jobs
```

without managing infrastructure.

---

AWS handles:

```text
Provisioning, Scheduling, Scaling, & Termination
```

---

You focus on:

```text
The Job
```

not the servers.

---

# What is a Batch Job?

A job that:

```text
Starts, Runs, & Finishes
```

---

Examples:

```text
Image Processing, Video Encoding, Financial Simulations, Scientific Computations, & Data Analytics
```

---

Not:

```text
Continuous Web Servers

Streaming Applications
```

---

# Example

```text
1:00 AM  Start Processing 10,000 Images

↓

3:00 AM  Finished
```

---

Job completed. Resources removed.

---

# Core Architecture

```text
Submit Job AWS Batch → Compute Environment → ECS / EKS / Fargate → Docker Container → Results
```

---

# Docker-Based Execution

Every Batch Job typically runs:

```text
Docker Image
```

---

Example:

```dockerfile
FROM python:3.12

COPY app.py .

CMD ["python","app.py"]
```

---

AWS Batch executes:

```text
Container not Virtual Machine
```

---

# Compute Environments

Batch needs compute power.

---

Options:

```text
EC2, Spot EC2, ECS, EKS, & Fargate
```

---

AWS provisions resources automatically.

---

# EC2 Compute Environment

AWS launches:

```text
EC2 Instances
```

---

Best for:

```text
Large Workloads & Maximum Control
```

---

# Spot Compute Environment

AWS launches:

```text
Spot Instances
```

---

Benefits:

```text
Huge Cost Savings
```

---

Ideal for:

```text
Fault-Tolerant Jobs
```

---

Exam keyword:

```text
Reduce Batch Cost
```

Answer:

```text
Spot Instances
```

---

# Fargate Compute Environment

No EC2 management.

---

Architecture:

```text
Batch → Fargate → Container
```

---

Serverless.

---

Best for:

```text
Small to Medium Jobs
```

---

# EKS Compute Environment

```text

```

---

Useful when company already uses:

```text
Amazon EKS
```

# Batch vs Lambda

Most important exam comparison.

---

# Lambda

Best for:

```text
Short, Event-Driven, & Serverless Tasks
```

---

Examples:

```text
S3 Trigger, API Request, & SNS Event
```

---

Limit:

```text
15 Minutes
```

---

# Batch

Best for:

```text
Long Running Jobs
```

---

Examples:

```text
Machine Learning, Video Encoding, Simulation, & Large ETL
```

---

Limit:

```text
No Practical Limit
```

---

# Comparison Table

| Feature | AWS Batch | AWS Lambda |
|----------|----------|----------|
| Time Limit | No limit | 15 min |
| Runtime | Any Docker Image | Supported runtimes |
| Compute | EC2/Fargate/EKS | Serverless |
| Storage | EBS, EC2 | Limited temporary storage |
| Scaling | Massive workloads | Event-driven |
| Best For | Batch jobs | Short functions |

# DevOps Interview Insight

If asked:

"Why would you choose AWS Batch instead of Lambda?"

A strong answer:

1. Lambda has a 15-minute execution limit.
2. Batch supports long-running workloads.
3. Batch can use custom Docker images.
4. Batch can scale to thousands of concurrent jobs.
5. Batch can leverage Spot Instances for major cost savings.
6. Batch is ideal for ETL, scientific computing, ML training, and media processing.
